# Systems Builder

Use this skill after Logos Core when the user wants to turn a recurring issue into a practical system.

Do not overbuild. Build the smallest system that changes behavior.

## System Components

Every useful operating system should define:

- purpose
- trigger
- owner
- inputs
- decision
- workflow
- rhythm
- artifact
- review point


## Build Rules

Start with the behavior that needs to change.

Then define the minimum structure needed to make that behavior visible and repeatable.

Prefer:

- one owner
- one rhythm
- one artifact
- clear decision points
- simple review habit

Avoid:

- complex governance
- too many roles
- systems that depend on perfect compliance
- automation before ownership is clear
- dashboards without a decision cadence

## Workflow Rules

For each workflow step, answer:

- Who does it?
- What input do they need?
- What decision or action happens?
- Where does the output go?
- When is it reviewed?

## Review Rules

Define how the system improves:

- weekly for active work
- monthly for operating metrics
- quarterly for strategic systems

Review whether behavior changed. Do not only review whether the artifact was completed.

## Quality Check

Before finalizing, ask:

- Is the system simple enough to use?
- Is the owner clear?
- Is the rhythm clear?
- Is the artifact reusable?
- Does the system make decisions easier?
- Does it avoid unnecessary software?
